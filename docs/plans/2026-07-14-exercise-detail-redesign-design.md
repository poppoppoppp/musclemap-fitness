# 动作详情页重构设计

## 目标

将现有动作详情页重构为移动端优先的训练入口。用户按“动作示意、关键提示、问题排查、详细信息、记录或加入训练”的顺序完成任务，同时保留现有动作 ID、训练状态、姿态方案上下文和替代动作跳转能力。

## 架构

- 保留唯一详情路由 `/exercises/:exerciseId`，不创建平行页面。
- 在现有 `Exercise` 类型上增加可选详情字段。只有 `one-arm-dumbbell-row` 提供完整示例，其他动作通过统一解析函数从现有 `steps`、`cues`、`commonMistakes`、`tags` 和 `alternatives` 安全降级。
- 页面拆为媒体示意、关键提示、问题排查、次级入口、抽屉内容和固定操作栏等中等粒度组件。
- 所有抽屉复用 `SnapBottomSheet`。记录和加入训练复用 `activeWorkout` 工具函数，不创建独立记录系统。
- 收藏当前没有可复用状态，使用一个只保存动作 ID 集合的本地存储工具。
- 静态动作图路径集中由动作数据或媒体工具生成，目录为 `public/exercise-media/<exerciseId>/start.webp` 与 `peak.webp`。加载失败显示统一的无文字占位图形。

## 交互

- 顶栏提供返回与收藏，返回目标继续尊重 `from=workout` 和姿态方案查询参数。
- 问题卡点击后打开单项问题抽屉，“查看更多”打开全部问题抽屉。
- 动作说明、训练部位、替代动作分别打开抽屉。替代动作链接进入既有详情路由。
- 有当前训练且未加入时，主按钮加入动作并原位显示成功反馈；已加入时显示“返回当前训练”；无当前训练时显示“添加到训练”，复用现有创建训练并加入动作流程。
- “记录一次”确保动作在当前训练中，然后跳转 `/workout-log?focusExercise=<activeExerciseId>`，直接复用训练记录界面。

## 视觉与响应式

- 页面使用接近黑色背景、深灰卡片、低对比边框和荧光绿训练强调色，保持现有系统字体。
- 内容最大宽度受控并居中。移动端单列，媒体区域始终优先双列；320px 下缩小间距和字号，不转为三行提示。
- 固定操作栏包含安全区，并为正文预留足够底部空间。详情路由隐藏全局底部导航。
- 图标统一使用轻量线性 SVG，红色仅出现在问题占位图的异常标记中。

## 兼容与错误处理

- 所有新增字段可选，解析器提供非空标题、说明、提示、问题与媒体路径，不渲染 `undefined` 或“暂未配置”。
- 旧动作图片请求失败后切换到统一占位，不影响布局。
- 姿态动作继续保留方案上下文，并将其并入动作说明抽屉。
- 加入动作继续使用既有重复检测；收藏本地存储读取失败时回退为空集合。

## 验证

- Playwright 覆盖完整示例、旧动作降级、收藏、四类抽屉、三种训练按钮状态、记录跳转、替代动作跳转、底栏隐藏与 320/390px 无横向溢出。
- 运行 TypeScript/Vite 构建、现有 Playwright 测试，并用浏览器截图检查移动端和桌面端布局。
- 只修改本地文件，不执行 Git 提交、推送或部署。
