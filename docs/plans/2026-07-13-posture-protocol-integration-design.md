# 体态改善方案接入设计

## 目标

将恢复数据中的可发布体态改善方案作为有序动作模块加入当前训练。方案在添加动作抽屉内浏览，动作详情复用现有页面，当前训练和历史使用独立快照，旧数据继续可读。

## 已确认边界

- `postureDataset.v0.1.json` 是唯一内容真源，不补写缺失处方或 metadata-only 内容。
- V0.1 只发布 `SHOULDER_001`。`conditionally_ready`、`metadata_only` 和 `hold` 内容保持隐藏。
- 体态改善不是独立训练模式，不新增路由入口或底部导航。
- 只支持整套加入；加入后可以沿用现有动作编辑、记录、完成和删除能力。

## 方案比较

### 方案 A：顶层方案快照关联现有动作实例（采用）

在 `ActiveWorkout` 和 `WorkoutLog` 增加可选的体态方案快照数组，方案通过动作实例 ID 关联现有扁平动作。动作实例增加可选方案上下文快照。

优点：旧数据天然兼容；现有组数、次数、备注、完成和归档逻辑可复用；分组名称、标签和来源不会随数据集更新而变化。缺点：渲染时需要把方案元数据和动作实例组合。

### 方案 B：把动作完整嵌套到方案中

将 `ActiveWorkout.exercises` 改造成普通动作与方案联合类型。

优点：结构直观。缺点：会重写当前训练、归档、历史、统计和备份的大量路径，兼容风险高，不符合最小改动原则。

### 方案 C：仅根据动作 ID 在 UI 中推导分组

保持数据结构不变，仅在训练页面按当前数据集重建方案。

优点：代码量少。缺点：数据集更新会改变旧训练和历史；无法可靠表示用户修改；不满足快照要求。

## 数据设计

- `PostureProtocolWorkoutSnapshot` 保存实例 ID、源方案 ID、名称、体态标签、加入时间、外层顺序、修改状态和动作实例 ID。
- `PostureExerciseWorkoutContext` 保存在动作实例上，保存动作名称、方案顺序、处方、作用和专项提示。
- `ActiveWorkoutExercise.source` 增加 `posture`；新增字段均可选，旧 localStorage 无需迁移。
- 归档时复制方案快照和动作上下文到 `WorkoutLog`，不再引用数据集对象。

## 数据过滤

`isProtocolVisibleInApp` 是唯一发布过滤入口，依次检查：

1. `status === "ready"`；
2. 方案 ID 位于集中发布白名单；
3. 至少存在一个 `includeByDefault !== false`、非 hold 且能解析标准动作的条目。

V0.1 发布白名单只包含 `SHOULDER_001`。体态问题、方案数量和列表全部从过滤后的方案动态生成。

## 交互设计

- “2D 找肌群”和“体态改善”在添加动作抽屉头部并列，同一视觉规格。
- 抽屉内部使用动作列表、体态问题、方案列表和方案详情四级视图，不叠加新抽屉。
- 仅一套方案的问题直接进入详情；多套时显示方案列表。
- 方案详情展示真实摘要、标签、动作、理论和来源折叠区；吸底按钮显示将添加的动作数。
- 加入后关闭抽屉、滚动到新分组并显示成功状态。重复点击在处理期间禁用。
- 训练中分组可折叠、整体删除、外层上下移动；内部动作可上下移动。任何内部编辑、删除或重排都会标记“已修改”。

## 动作详情与返回

体态标准动作转换为现有 `Exercise` 详情模型的只读视图，但不加入普通动作目录。详情页额外显示“本方案中的安排”。查询参数记录来源：普通动作库返回动作库，训练中返回训练中，方案返回原方案详情并恢复抽屉滚动位置。

## 验证

- 直接导入工具函数的 Playwright 测试覆盖过滤、去重、排序、快照独立性、修改标记、旧数据和空数据。
- 浏览器流程覆盖抽屉层级、详情复用、加入、刷新、分组修改/删除、320px 和 390px 溢出。
- 最终运行目标测试、全量 `npm test` 和 `npm run build`，并在浏览器检查主要移动端状态。
