# MuscleMap 全量体态改善数据接入设计

## 目标

在不改变普通训练扁平 `exercises` 主流程的前提下，将附件中的 12 条方案、32 个稳定动作引用、4 个观察项、2 条理论素材和 1 条观念引导完整接入现有体态改善抽屉，并保持上一版 localStorage、训练历史和动作详情兼容。

## 数据边界

- 保留 `postureDataset.v0.1.json` 不变，新建 `postureDataset.v0.2.json` 作为当前唯一内容真源。
- v0.2 显式存储分类、动作家族、观察项、标准动作、方案步骤、理论和观念引导。
- 方案步骤使用任意 `groupKey/groupLabel`，不将阶段固定为少数枚举。
- 缺失剂量保持缺失；来源画面疑似剂量保留 `source/confidence`；`sourceClaims` 不进入默认用户文案。
- `visualReviewRequired` 只标记附件明确要求复核且现有详情无法补足的动作。

## 动作去重

- 方案始终引用附件给出的稳定 `EX_*` ID。
- `EX_GLUTE_BRIDGE` 和 `EX_CONTRALATERAL_DEAD_BUG` 通过 `libraryExerciseId` 复用普通动作库内容。
- 上一版小写体态动作 ID 通过 `legacyIds` 映射到新的标准动作，旧活动训练仍可恢复。
- 同一动作跨方案只定义一次；难度或器械变式通过 `familyId/variantOf` 关联。
- `SHOULDER_002` 是唯一需要用户二选一的动作家族，不扩展为通用的任意步骤选择器。

## 浏览与详情

- 保留“添加动作 → 体态改善”的入口和单抽屉结构。
- 首层展示 8 个分类及真实方案数量，并在顶部内嵌观念引导。
- 第二层展示分类内方案；第三层按 `groupLabel` 展示全部阶段。
- 观察项使用独立视觉卡片，不提供动作详情链接，也不计入可训练动作数。
- 可选动作显示“可选”且默认不加入；缺失剂量显示“剂量未说明”；低置信度剂量显示来源提示但不作为自动训练处方。
- 动作继续进入现有 `ExerciseDetail`，方案上下文与公共动作说明分开。

## 训练、快照与历史

- 加入方案时，仅 exercise 步骤进入现有扁平 `ActiveWorkout.exercises`；observation 只进入方案步骤快照。
- `PostureProtocolWorkoutSnapshot` 保留原字段并新增可选 `stepSnapshots`，因此旧数据无需迁移。
- 每个 exercise 快照保留阶段、方案剂量、置信度和用户可见注意事项；源数据对象不被训练实例引用。
- 明确的持续时间通过现有 set 结构的可选 `durationSeconds` 记录；缺失剂量不自动生成组数、次数或时长。
- 训练完成数量只统计真实 exercise，历史从快照显示阶段与观察项；成长统计忽略没有力量次数的时长记录。

## 校验与错误处理

- `validatePostureDataset` 返回结构化错误，并在开发环境导入数据时失败可见。
- 校验数量、重复 ID、引用完整性、动作家族、缺失剂量、危险文案和观察项统计边界。
- 不可解析的旧快照继续走现有防御性读取；无效新方案不进入用户列表。

## 验证

- 数据测试覆盖附件列出的 16 项自动校验。
- Playwright 覆盖分类、阶段、观察项、变式选择、方案加入、刷新、历史和普通动作回归。
- 在 320px 与 390px 检查横向溢出、固定操作区和长阶段列表。
- 最终运行 `npm test`、`npm run build` 和 `git diff --check`。
